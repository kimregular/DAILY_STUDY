# 예외처리

자바와 거의 동일하다. 하지만 차이가 하나 있다. 기본적인 예외 던지는 방식을 알아보자.

```kotlin
if(str.length > 10) {
	throw IllegalStateException("str($str) is too long")
}
```

- 자바와 유사하지만 new 키워드가 불필요하다.

## @Transactional 주의사항

Spring의 `@Transactional`은 기본적으로 `RuntimeException`, `Error` 같은 Unchecked Exception이 발생했을 때 롤백된다.  
반면 `IOException`, `SQLException` 같은 Checked Exception은 기본 설정에서는 롤백 대상이 아니다.

### 1. Java - Checked Exception은 컴파일러가 처리를 강제함

Java에서는 Checked Exception이 발생할 수 있는 코드를 작성하면 컴파일러가 `catch` 또는 `throws` 처리를 강제한다.

```java
// 자바
@Transactional
public void process() throws IOException {
    throw new IOException();
}
```
위 코드는 예외가 밖으로 던져지더라도 `IOException`이 `Checked Exception`이기 때문에 기본적으로 트랜잭션이 롤백되지 않는다.

`Checked Exception`에서도 롤백이 필요하다면 `rollbackFor`를 명시해야한다.

```java
@Transactional(rollbackFor = IOException.class)
public void process() throws IOException {
    throw new IOException();
}
```

### 2. Kotlin - Checked Exception 처리를 강제하지 않음

Kotlin은 `Checked Exception` 개념을 언어 차원에서 강제하지 않는다.  
따라서 Java API에서 `IOException` 같은 `Checked Exception`이 발생할 수 있어도 컴파일러가 `try-catch`나 `throws` 처리를 요구하지 않는다.

```kotlin
@Transactional
fun process() {
    throw IOException()
}
```

이 경우 `IOException`은 발생하지만 Spring 기본 롤백 정책상 Checked Exception이므로 롤백되지 않을 수 있다.

따라서 Kotlin에서 `Checked Exception`이 발생할 수 있는 작업을 트랜잭션 안에서 수행한다면, 롤백 정책을 명확히 지정하는 것이 안전하다.

```kotlin
@Transactional(rollbackFor = [IOException::class])
fun process() {
    throw IOException()
}
```

또는 서비스 계층에서는 `Checked Exception`을 비즈니스 `RuntimeException`으로 변환해서 던지는 방식도 사용할 수 있다.

```kotlin
@Transactional
fun process() {
    try {
        // IOException 발생 가능 작업
    } catch (e: IOException) {
        throw IllegalStateException("파일 처리 실패", e)
    }
}
```
